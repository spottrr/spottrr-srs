\section{System Architecture Overview}
\label{section:architecture-overview}

Figure~\ref{fig:system-architecture-highlevel} shows the high-level system architecture, illustrating the relationship between the frontend, backend, AI pipeline, and storage layers.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.85\textwidth]{assets/high-level-architecture.png}
    \caption{High-level system architecture.}
    \label{fig:system-architecture-highlevel}
\end{figure}

The system consists of four main layers:

\begin{itemize}
    \item \textbf{Frontend (React).} The user-facing web application. Users upload videos, enter search queries, browse face clusters, and view results. It communicates with the backend exclusively through HTTP requests.

    \item \textbf{Backend (FastAPI + Celery).} The central coordinator. The REST API receives upload and search requests, validates inputs, and routes them to the appropriate service. For uploads, it dispatches processing jobs to a Task Queue (Celery + Redis) that runs the AI Pipeline asynchronously, allowing users to continue using the application while their video is being indexed.

    \item \textbf{Storage (PostgreSQL + File Storage).} PostgreSQL with pgvector stores all object region embeddings and face embeddings with metadata (timestamps, bounding boxes, source camera) and handles vector similarity search. File Storage holds the original video files for playback when users click on a search result.

    \item \textbf{AI Pipeline (SAM + SigLIP2 + Face Similarity).} The core intelligence of the system. It operates in two modes: during ingestion, it segments objects (SAM), encodes object regions and whole frames (SigLIP2), applies rule-based deduplication to reduce duplicate embeddings across video frames, and detects faces for identity-based search. During search, it encodes the user's query into the same vector space for similarity matching.
\end{itemize}

\textbf{(a) Ingestion Pipeline.} Uploaded video is split into keyframes at 1--5 FPS. Each frame goes through two parallel paths. SAM \cite{kirillov2023segment} segments every object in the frame into regions with context padding. SigLIP2 \cite{zhai2025siglip2} then encodes each object region and the whole frame into vector embeddings. A rule-based deduplication process (see Section~\ref{subsubsec:dedup}) filters near-identical frames and groups tracked object embeddings, keeping only the best representative per object to reduce index bloat. In parallel, face detection identifies and encodes faces. All embeddings are stored with timestamps in the vector database.

\textbf{(b) Search Pipeline.} The user's text, image, or face query is encoded into the same vector space. The system performs cosine similarity search against all stored embeddings. A hybrid scoring function then computes each candidate's final score as the maximum of its best object region score and whole-image score, ensuring small objects detected by SAM are not lost. Results are ranked and the corresponding frames are retrieved for display.

\section{Software Design}
\label{section:software-design}

This section describes the runtime behavior of the system through sequence diagrams for the two most important use cases.

\subsection{Video Upload and Indexing (UC-4)}

The user uploads a video file through the Web App. The REST API validates the file and stores it in File Storage, then enqueues a processing job in the Task Queue. The user is free to continue using the application while processing runs in the background. The Task Queue extracts keyframes at 1--5 FPS, then runs two parallel paths: SAM segments each frame into object regions which are encoded by SigLIP2 into vector embeddings, while face detection identifies and encodes faces. A rule-based deduplication process filters near-identical frames and groups tracked object embeddings, keeping the best representative per object. All embeddings are stored in the Vector DB with timestamps and metadata. Figure~\ref{fig:seq-upload} shows this sequence of interactions.

\subsection{Text Search (UC-1)}

The user submits a text query through the Web App. The REST API sends the query to the AI Pipeline, where the image-text encoder (SigLIP2) encodes it into a vector embedding. The system then performs cosine similarity search against the Vector DB. A hybrid scoring function computes each candidate's final score as the maximum of its best object region score and whole-image score. \section{AI and Data Design}
\label{section:ai-data-design}

\subsection{Data Sources and Collection}

The system processes user-uploaded video footage as its primary data source. The following public dataset is used for development and evaluation:

\begin{itemize}
    \item \textbf{COCO val2017.} Image dataset with bounding box annotations across 80 object categories (1,000 images, 66 queries across 5 categories). Used for model selection benchmarking and retrieval evaluation. Task: given a text query, retrieve the matching image.
\end{itemize}

\subsection{Model Design}

The retrieval pipeline consists of two core components: a segmentation model and a vision-language encoder. To select the best combination, we conducted a preliminary benchmark on a subset of COCO val2017 (1,000 images, approximately 30 images per query category, 66 queries across 5 categories). This benchmark is not the final system evaluation; it serves as a model selection experiment to inform our architecture decisions. The benchmark task is image-to-text retrieval: given a text query, find the matching image.

\subsubsection{Segmentation Model}

We use SAM (Segment Anything Model) \cite{kirillov2023segment} for class-agnostic segmentation. SAM segments every object in each frame without requiring a text prompt, which is essential for our offline indexing approach. Unlike text-prompted segmentation models such as Grounding DINO \cite{liu2023groundingdino} or OpenWorldSAM \cite{xiao2025openworldsam}, SAM does not need to know what the user will search for at indexing time. This allows the system to process footage once and handle arbitrary queries at search time.

\subsubsection{Vision-Language Encoder Selection}

We evaluated three vision-language encoders on whole-image retrieval to determine which produces the strongest embeddings. Table~\ref{tab:encoder-selection} summarizes the results.

        \toprule
        \textbf{Encoder} & \textbf{Recall@5} & \textbf{Precision@5} & \textbf{Small Obj. R@5} \\
        \midrule
        BLIP \cite{li2019blip} & 0.027 & 0.027 & 0.022 \\
        CLIP \cite{radford2021learning} & 0.480 & 0.480 & 0.244 \\
        \midrule
        \textbf{SigLIP2} \cite{zhai2025siglip2} & \textbf{0.640} & \textbf{0.640} & \textbf{0.411} \\
        \bottomrule
    \end{tabular}
    \caption{Vision-language encoder comparison on COCO val2017 (1,000 images, simple + small object queries, $\sim$30 objects per image). SigLIP2 outperforms CLIP and BLIP across all metrics.}
    \label{tab:encoder-selection}

SigLIP2 achieves the highest scores on all metrics, making it the initial selected encoder. However, all whole-image encoders struggle with small and background objects (rightmost column), motivating the addition of SAM segmentation.

\subsubsection{Effect of SAM Segmentation}

We then paired SAM with each encoder using our hybrid scoring approach, where the final score for each image is the maximum of the best object region score and the whole-image score. Table~\ref{tab:sam-effect} shows the results.

\begin{table}[H]
    \centering
    \begin{tabular}{lccc}
        \toprule
        \textbf{Pipeline} & \textbf{Recall@5} & \textbf{Precision@5} & \textbf{Small Obj. R@5} \\
        \midrule
        SAM + CLIP & 0.644 & 0.640 & 0.519 \\
        \midrule
        \textbf{SAM + SigLIP2} & \textbf{0.764} & \textbf{0.760} & \textbf{0.619} \\
        \bottomrule
    \end{tabular}
    \caption{Effect of adding SAM segmentation. Both encoders improve, with SAM + SigLIP2 achieving the best results. Small object retrieval improves from 0.411 (SigLIP2 alone) to 0.619.}
    \label{tab:sam-effect}
\end{table}

Adding SAM segmentation improves both encoders, and the stronger encoder (SigLIP2) benefits more. Based on these results, we initially selected SAM + SigLIP2 as the retrieval backbone. The architecture is encoder-agnostic; the vision-language encoder can be swapped without changing the rest of the pipeline.

\subsubsection{Rule-Based Deduplication for Video}
\label{subsubsec:dedup}

The region-level approach described above works well for static images, but video introduces a new challenge. When SAM tracks an object across consecutive frames, the same person walking through a 5-minute clip might appear in 300 frames, producing 300 near-identical embeddings in the index. This causes three problems: the index grows unnecessarily large, queries slow down, and search results are filled with duplicates of the same object.

To address this, the system uses a two-stage rule-based deduplication approach:

\begin{enumerate}
    \item \textbf{Frame-level filtering.} Each frame is encoded by the vision-language model and compared against the previous frame's embedding. If the cosine similarity exceeds a threshold, the frame is considered near-identical and skipped. This reduces the number of frames that need to be processed by SAM, saving computation time.

    \item \textbf{Object-level deduplication.} For the remaining frames, tracked object embeddings are compared within a sliding window (e.g., 5 frames with 3-frame overlap). If the embeddings of the same tracked object across frames are sufficiently similar, only the best representative is kept, selected based on the highest SAM confidence score. This ensures each unique object appearance is stored once rather than hundreds of times.
\end{enumerate}

This rule-based approach requires no training data or learned weights. It extends region-based retrieval from static images to video in a simple and practical way. Existing work like SAM-CLIP Search \cite{mulani2025samclip} operates only on static images; this deduplication mechanism is what makes the pipeline practical for real surveillance footage where objects move, get occluded, and appear across hundreds of frames.

\subsection{Evaluation}

The system will be evaluated on two levels:

\begin{enumerate}
    \item \textbf{Retrieval quality.} Recall@5, Precision@5, and F1@5 on COCO val2017, comparing the full pipeline (SAM + SigLIP2) against whole-image baselines (CLIP, SigLIP2, BLIP) and segmentation baselines (SAM + CLIP). Query categories include simple objects and small/background objects (objects occupying less than 10\% of image area). The Precision-Recall tradeoff will be analyzed to find the optimal operating point.
    \item \textbf{End-to-end system.} Query latency, indexing throughput (frames per second), and storage efficiency (embeddings per video minute). User testing with real surveillance footage to assess practical usability.
\end{enumerate}
